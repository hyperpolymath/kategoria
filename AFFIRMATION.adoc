== What this is, and how it is designed to work

*What it is.* A short, dated, jointly-signed snapshot of what we can _honestly
and verifiably_ claim about this repo at one exact commit. Nothing here is
marketing and nothing is a promise about the future — both of those live in the
README. This file is the receipt.

*How it is designed to work.* Three moving parts make it trustworthy:

. *Ground truth, not memory.* Every claim below was produced by _running the
  tool in the session that wrote this file_ — here, `idris2 --check` against the
  actual route and example files. Status docs (ROADMAP, EXPLAINME, PROOF-NEEDS)
  are NOT evidence; where they contradict the live run, *the live run wins and
  we say so* (and they do contradict it — see <<state>>).
. *A frozen anchor.* The file names the exact commit SHA, branch, UTC timestamp,
  and toolchain (see <<anchor,Verifiable anchor>>), so "true" always means "true
  _at this point_". Move the SHA and the file becomes a draft until re-run.
. *A real signature.* It is landed by a *signed git commit*; that signature over
  this content at the anchored SHA is what makes the affirmation tamper-evident
  and attributable — not the prose alone.

*We are fallible.* We can be wrong, stale, or simply mistaken about our own
work. This file is our best honest belief, not a proof of its own correctness.
Treat it as a falsifiable claim, not gospel.

== The epistemic contract (read this before you trust _or_ attack)

This document records our *best joint belief* at the timestamp below. It is
*not a guarantee of correctness.* We may be wrong. We may have missed something
— a file that type-checks for the wrong reason, a stale status table, an
encoding we believe demonstrates a level but that a type theorist would reject.

*No intentional overclaim.* That is the only guarantee here: we have not
_knowingly_ inflated anything. Where a level *fails to type-check* we say so.
Where a "✓" in a status doc is *refuted by the checker*, we say so. Where a
level is an honestly-documented *wall* rather than a solved problem, we say so.
If you find an honest claim here that turns out false, that is an error to be
fixed — not a lie.

*Standing invitation to refute.* You are _invited_ to bulldoze any claim in this
file. We want that — and for a taxonomy/challenge repo, "tell us where we are
wrong about the maths" is _the entire point_. But the fair form of the attack
is:

. Read this file at the stamped commit.
. Reproduce (or fail to reproduce) the checks in <<reproduce,Reproduce it yourself>>.
. _Then_ tell us where the discrepancy is, against the artefact as it stood at
  this moment — so we can either justify it or concede it on the record.

An attack that skips steps 1–2 is attacking a strawman of a different date.

*No fights before the facts are cleared up.* We are not interested in a dispute
over a claim until the discrepancy has actually been checked against the
artefact at this commit and we have had the chance to either justify it or
concede it on the record. Good faith both ways: bring a reproducible
discrepancy and we will fix the claim, the doc, or the code — promptly and
without defensiveness. The goal is a corrected record, not a won argument.

=== The honest nuance you must not lose (proved-core vs tested-impl)

This is a taxonomy/challenge repo, so the gap takes a specific shape:

* *Nothing here is "proved" in the proof-checker sense.* A file type-checking
  under `idris2 --check` means *Idris2 accepted it as well-typed* — that is
  "type-checks" / "tested", not a discharged theorem. There is no `.ipkg`/`pack`
  build, no test runner, no proof obligation that a checker reports green. The
  word *proved* is therefore not earned anywhere in this repo at this commit.
* *L10 type-checks but is explicitly a documented wall, not a cubical proof.*
  `Level10_CubicalTypes.idr` carries its own header `STATUS: ✗ IMPOSSIBLE on
  this route (QTT core ≠ cubical core)` and spells out that univalence/funext
  would need a non-computing `postulate` or `believe_me` in Idris2's QTT core.
  It compiling is _consistent with_ the README's honest "L10 is the wall" stance;
  do *not* read a green L10 as "cubical types achieved".
* *L09 contains a `postulate recvPlaceholder`* (a deliberate stub the file
  itself flags) — and the file *does not even type-check* at this commit anyway
  (the `private`/`postulate` block is a parse error under Idris2 0.8.0).
* *The "settled core" the repo demonstrates (L1–L6 type theory) is not the same
  as "the kategoria language implements all ten levels".* Six type-checking
  files are six demonstrations, not an implemented language and not a proof of
  the challenge's central question.

== How the plan is working out (discoveries)

* *What turned out "magical" (a believed-answer that dissolved on contact).*
  The status docs (ROADMAP "9/10", EXPLAINME "ten level files ... all type-check")
  read as if Route α were nearly complete. On contact with `idris2 --check`, four
  of the ten files do not compile and the build harness that was supposed to
  prove this (`just test-alpha`) does not exist. The "9/10" was a believed score,
  not a checked one. That is the gap this whole AFFIRMATION format exists to catch.
* *What a check made newly clear.* Running the checker turned a fuzzy "early
  implementation" into a precise, defensible number — *6/10 files type-check,
  named and reproducible* — which is a far more useful starting point for the
  next session than any colour-coded matrix, and is the honest figure the README
  matrix (all `.`) should graduate toward.

== Reproduce it yourself

From the repo root, at the commit above, with Idris2 0.8.0 on `PATH`:

[source,sh]
----
# Type-check all ten Route α level files (the real ground truth):
for f in routes/alpha-extend/Level0*.idr routes/alpha-extend/Level10*.idr; do
  echo "== $f =="; idris2 --check "$f"; echo "exit $?"
done
# Expect: L01,L02,L04,L05,L06,L10 -> exit 0 ; L03,L07,L08,L09 -> exit 1  (6 pass / 4 fail)

# Type-check the three example demos:
for f in examples/Demo_*.idr; do echo "== $f =="; idris2 --check "$f"; echo "exit $?"; done
# Expect: all three exit 0

# Note: `just test` / `just build` are RSR-template TODO stubs (echo only);
# `just test-alpha` / `just test-all` / `just progress` referenced by EXPLAINME
# DO NOT EXIST in the Justfile at this commit.
----

pass; L3,L7,L8,L9 fail); all 3 example demos type-check; no proof-checker or test
harness runs.*
